Introduction

This site contains all the information about the project for the course "Markup Languages". The topic is about an application's settings. In my example, I used an application called "Strava" which is basically a social sport activity tracker. The project consists of an XML file which describes the settings of the application. There is also an XML Schema file for validation purposes and and an XSLT file for transformation of the XML to JSON format. Finally, the JSON output file is also included.

Methodology

As mentioned earlier the methodologies that are used are the following:

Technology Purpose
XML As a description language, i used XML in order to organize the data as clear as possible
XML Schema The main purpose of this technology is to check if the XML file is organized well and according to the rules
XSL XSL is used for transformation of the XML file to other forms like HTML or JSON
JSON JSON, like XML is an easy way to display data in an organized way, easily readable by the user but also easy manageable by programming languages

Element description

Some basic elements that were used in the XML File are:

  • my_profile : This element contains tags that refer to profile settings. Some of them are: photos, name, birthday, gender, location, weight, url, bio. There are also sub elements with some attributes i.e. photo IDs and sizes.
  • my_account : This element contains tags that refer to account settings. Some of them are: member_since, membership_status, year, month, day. Some of these elements can only take specific values such as Free/Premium.
  • display_preferences : This element contains tags that refer to display preferences. Some of these elements are: units, default_sport, default_highlight_image. Again, some of these subelements must have a specific string value such as Kilometers and Kilograms/Miles and Pounds.
  • privacy : This element contains tags that refer to privacy settings. Some of them are: enhaced_privacy, flyby_visibility, log_visibility, hidden_locations, private_activities , hidden_from_leaderboards, data_usage. Most of these elements must have specific values such as on/off or Everybody/Nobody. The element hidden_locations contains sub elements of location type, each one of them having a specific ID as attribute.
  • email_notifications : This element contains tags that refer to email and notification settings. Some of them are: club_email, social_email, segment_email, newsletter, language. These elements also must have a value such as on/off or a language string.
  • my_gear : This element contains tags that refer to the user's gear. First sub elements are my_bikes and my_shoes. Each of these elements contain subelements of bike or shoe type with specific attributes such as ID, status. There are also sub elements like name and kilometers in each category.
  • my_apps: This elements contains sub elements of the type app. Each one of them is a specific app that is connected with the Strava account. Some attributes are app_id and access. The only sub element is the element name.
  • social_connections: Finally, this element contains all the services that are connected with the Strava account. It contains many sub elements with the type of service. There are some attributes like service_id and service_status that can only take specific values. There are two sub elements of each element service. These are name and icon. Icon element also has attributes such as id and size.
For practical reasons not every element is listed above.

Transformation description

Transformation was used to transform my XML document to a JSON one.

The XSL file contains some info and descriptions on how the file is supposed to transform. I use items like [] for tables,{} for objects and "":"" for element-value combos. We can access element or attribute content using xsl:value-of select="". I use xsl:for-each to loop into elements with many subelements if the same type. I also use xsl:if test="position() != last()" after every loop in order to automatically decide if a comma(,) is needed.
